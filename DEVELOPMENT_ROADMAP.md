# Business Universe — этапы разработки до релиза

Этот файл — основной roadmap для последовательной разработки игры с помощью Codex, Claude Code и других coding-агентов.

Игра должна развиваться не как бесконечный набор несвязанных ресурсов, а как понятная экономическая стратегия/idle-tycoon с производственными цепочками, населением, рабочей силой, потребностями, развитием классов и монетизацией через Яндекс Игры.

Вдохновение системой населения можно брать из Anno: сначала простые жители, затем более развитые социальные классы с новыми потребностями и профессиями. Нельзя копировать названия, интерфейс, баланс, ассеты, тексты или конкретные механики Anno один в один. Нужна собственная реализация в стилистике Business Universe.

---

## 1. Главная цель проекта

Создать браузерную idle/tycoon-игру, которая:

- понятна в первые 30–60 секунд;
- даёт игроку первую завершённую производственную цепочку за 5–10 минут;
- создаёт осмысленный выбор между продажей сырья и глубокой переработкой;
- удерживает игрока за счёт населения, дефицита рабочей силы, потребностей, контрактов, открытий и offline-прогресса;
- корректно работает на мобильных устройствах и desktop;
- интегрирована с Яндекс Играми;
- монетизируется без принудительного pay-to-win;
- остаётся расширяемой: новые ресурсы, классы населения и здания добавляются через конфиги, а не через ветвления по id.

---

## 2. Приоритеты проекта

Порядок важности нельзя менять без отдельного решения владельца проекта.

### P0 — обязательно до первого публичного релиза

1. Исправить экономику и убрать доминирующую стратегию массовой покупки самого дешёвого здания.
2. Добавить строительные ресурсы и стоимость строительства в нескольких ресурсах.
3. Добавить население и ограничение производства рабочей силой.
4. Добавить минимум две полноценные производственные ветки помимо картофеля.
5. Добавить потребности населения и развитие классов.
6. Добавить offline-прогресс с безопасным ограничением времени.
7. Добавить адаптер Яндекс Игр, Loading API, рекламу и облачные сохранения.
8. Добавить unit-тесты критической доменной логики и CI.
9. Удалить или скрыть все неработающие кнопки и заглушки.
10. Провести мобильное тестирование, баланс первой сессии и проверку сохранений.

### P1 — сразу после стабильного MVP

1. Контракты и задания.
2. Достижения.
3. Ежедневные награды.
4. Улучшения зданий.
5. Простая система исследований.
6. Лидерборд.
7. Аналитика воронки и экономики.
8. Первый prestige/reset-цикл с постоянным бонусом.

### P2 — после появления реальных игроков и аналитики

1. События и временные контракты.
2. Дополнительные классы населения.
3. Новые регионы или экономические эпохи.
4. Коллекционные косметические элементы.
5. Расширенные исследования и специализации города.
6. Сезонный контент.

### Что пока не является приоритетом

- сложная 3D-графика;
- backend без конкретной необходимости;
- multiplayer;
- PvP;
- десятки языков до проверки удержания;
- большое количество ресурсов без связанных цепочек;
- дорогие визуальные эффекты, снижающие FPS;
- полный клон Anno или другой существующей игры.

---

## 3. Правила работы для Codex / Claude Code

Каждый агент обязан соблюдать следующий протокол.

### Перед началом задачи

1. Прочитать `AGENTS.md`, `CLAUDE.md`, `README.md` и этот файл.
2. Изучить текущую реализацию, а не ориентироваться только на roadmap.
3. Выбрать только один этап или одну законченную подзадачу.
4. Кратко описать план и затрагиваемые файлы.
5. Проверить, влияет ли изменение на сохранения.
6. Проверить, нужны ли новые i18n-ключи.

### Во время работы

- Не переписывать работающие модули без необходимости.
- Не менять баланс вне текущей задачи.
- Не добавлять зависимости без явной причины.
- Не смешивать доменную логику с React-компонентами.
- Не добавлять проверки вида `if (building.id === '...')` для обычных зданий.
- Не хранить производные значения в сохранении.
- Не обращаться к SDK Яндекс Игр из компонентов или domain.
- Не допускать частичного списания ресурсов.
- Не терять готовую продукцию при полном складе.
- Не сохранять игру на каждом кадре или каждом simulation tick.
- Не ломать существующие сохранения без миграции версии.

### После работы

Запустить:

```bash
npm run typecheck
npm run lint
npm run format:check
npm run i18n:check
npm run build
```

Если добавлен test runner, также запустить:

```bash
npm test
```

В отчёте указать:

- что изменено;
- какие сценарии проверены;
- какие команды реально прошли;
- была ли изменена схема сохранений;
- какие ограничения остались;
- что не входило в задачу и не было изменено.

После выполнения подзадачи агент должен остановиться и не брать следующий этап самостоятельно.

# ЭТАПЫ РАЗРАБОТКИ

---

## Этап 0. Зафиксировать безопасную техническую базу

**Приоритет: P0**

### Цель

Перед расширением экономики защитить текущую доменную логику от регрессий.

### Задачи

- Добавить unit-test runner, предпочтительно Vitest.
- Покрыть тестами чистые функции из `src/domain`.
- Добавить GitHub Actions для typecheck, lint, format check, i18n check, tests и build.
- Добавить тестовые фабрики состояния игры.
- Не тестировать визуальные детали на этом этапе.

### Обязательные тестовые сценарии

- ресурсы списываются атомарно;
- цикл не стартует при нехватке хотя бы одного входного ресурса;
- продукция не теряется при полном складе;
- продукция не выдаётся дважды после разблокировки склада;
- автопродажа начисляет правильную сумму;
- несколько циклов за один tick дают тот же результат, что несколько маленьких tick;
- отрицательный и бесконечный delta не изменяет игру;
- массовое строительство не покупает больше доступного;
- миграции v1 → current сохраняют прогресс;
- повреждённое сохранение безопасно восстанавливается.

### Критерий завершения

Критические правила экономики и сохранений покрыты тестами, CI запускается на каждом PR и коммите в `main`.

## Этап 1. Перестроить базовую экономику

**Приоритет: P0, самый первый продуктовый этап**

### Проблема

Фиксированная цена зданий приводит к бесконечному экспоненциальному росту и делает самое дешёвое здание лучшей стратегией на всей дистанции.

### Задачи

- Ввести рост цены каждой следующей копии здания.
- Хранить формулу цены в domain/config.
- Не вычислять стоимость в компонентах.
- Настроить окупаемость зданий по уровням прогресса.
- Сделать глубокую переработку выгоднее продажи сырья, но только при наличии логистики, рабочих и достаточного спроса.
- Добавить компактное форматирование больших чисел.
- Добавить dev-инструмент или чистую функцию для симуляции экономики на 5, 15, 30 и 60 минут.

### Рекомендуемая стартовая модель

```text
unitCost = baseCost * growthRate ^ ownedCount
```

Формулу нужно реализовать так, чтобы позже её можно было заменить без изменения UI и store.

### Балансовые цели

- первая ферма доступна сразу;
- первая переработка открывается в первые 3–7 минут;
- игрок не может бесконечно удваивать доход каждые 20 секунд;
- новая ступень должна быть заметно выгоднее предыдущей после настройки цепочки;
- нельзя делать одну ветку лучшей по всем параметрам;
- дефицит денег должен сменяться дефицитом ресурсов и рабочей силы.

### Критерий завершения

Игроку выгодно развивать цепочки, а не спамить только одним дешёвым зданием.

---

## Этап 2. Добавить строительные ресурсы

**Приоритет: P0**

### Цель

Здания должны строиться не только за деньги. Деньги остаются важными, но становятся лишь одной частью стоимости.

### Первые строительные ресурсы

- дерево;
- доски;
- камень;
- кирпич;
- инструменты.

Необязательно добавлять всё за одну задачу. Начать с дерева, досок и камня.

### Базовые цепочки

```text
Лесоруб → дерево
Лесопилка: дерево → доски
Каменоломня → камень
Глиняный карьер → глина
Кирпичный завод: глина + топливо → кирпич
Мастерская: доски + металл → инструменты
```

### Модель стоимости здания

```ts
constructionCost: {
  money: number;
  resources: ResourceAmount[];
}
```

### Правила

- Перед строительством проверяются деньги и все ресурсы.
- Списание происходит атомарно.
- При нехватке показывается точный список недостающих ресурсов.
- Массовое строительство рассчитывает максимально доступное количество с учётом всех ресурсов.
- Стоимость должна корректно масштабироваться для каждой следующей копии.
- Строительные ресурсы хранятся на складе и занимают вместимость.

### Критерий завершения

Постройка ключевых зданий требует подготовки строительной цепочки, а не только накопления денег.

---

## Этап 3. Население и рабочая сила

**Приоритет: P0**

### Цель

Производство должно зависеть от жителей. Здания без работников не производят товары.

### Первые классы населения

1. **Поселенцы** — базовый класс, выполняет простую сельскохозяйственную и добывающую работу.
2. **Ремесленники** — более развитый класс, работает на перерабатывающих предприятиях.

Названия могут быть изменены позже, но смысл должен сохраниться.

### Основные понятия

```ts
PopulationClassId = 'settler' | 'artisan'

PopulationState = {
  total: number;
  employed: number;
  available: number;
}
```

### Жилые здания

- дом поселенцев;
- улучшенный дом ремесленников.

Жилой дом:

- строится за деньги и строительные ресурсы;
- даёт ограниченное количество жителей;
- требует удовлетворения базовых потребностей;
- может улучшаться до следующего класса при выполнении условий.

### Рабочая сила зданий

Каждое производственное здание получает конфиг:

```ts
workforce: {
  populationClassId: PopulationClassId;
  required: number;
}
```

### Правила производства

- Рабочая сила закрепляется за активными зданиями.
- Если рабочих недостаточно, часть зданий простаивает.
- Нельзя допускать отрицательную доступную рабочую силу.
- Для массово построенного здания потребность умножается на `ownedCount`.
- Состояние нехватки рабочих должно быть машинно-читаемым, например `waiting_for_workers`.
- UI показывает требуемых, занятых и свободных жителей.
- Простое сырьевое производство использует поселенцев.
- Переработка и сложные фабрики используют ремесленников.

### Выбор модели распределения

Для первой версии использовать автоматическое распределение работников по фиксированному приоритету.

Позже можно добавить ручное включение/выключение зданий и приоритет производства.

### Критерий завершения

Игрок не может бесконечно покупать производственные здания без строительства жилья и развития населения.

---

## Этап 4. Потребности населения

**Приоритет: P0**

### Цель

Жители должны не только давать рабочую силу, но и потреблять товары, создавая внутренний спрос.

### Потребности поселенцев

Начальный вариант:

- еда: картофель или хлеб;
- базовая одежда;
- жильё;
- доступ к рынку.

### Потребности ремесленников

Начальный вариант:

- хлеб;
- обработанная еда;
- одежда;
- инструменты;
- улучшенное жильё;
- общественная услуга.

### Механика

- Потребности потребляют товары с определённым интервалом.
- Удовлетворённые потребности повышают счастье/стабильность.
- Нехватка товаров снижает эффективность или останавливает рост населения.
- Население не должно мгновенно исчезать при кратком дефиците.
- Должен быть понятный grace period.
- Потребление рассчитывается в domain.
- UI показывает текущий запас и прогноз времени до дефицита.

### Упрощение для MVP

Не вводить сложную симуляцию каждого жителя. Все расчёты вести агрегировано по классу населения.

### Критерий завершения

Производственные цепочки нужны не только для продажи, но и для поддержания и развития населения.

---

## Этап 5. Улучшение населения: поселенцы → ремесленники

**Приоритет: P0**

### Цель

Создать основной вертикальный прогресс игры.

### Условия улучшения дома

Пример:

- все базовые потребности поселенцев удовлетворены;
- есть строительные ресурсы;
- есть деньги;
- достигнут минимальный уровень города;
- есть доступ к рынку или общественному зданию.

### Результат улучшения

- часть поселенцев становится ремесленниками;
- дом меняет вместимость и набор потребностей;
- открываются новые производственные здания;
- уменьшается доступное число поселенцев;
- появляются новые дефициты рабочей силы.

### Важное правило

Улучшение населения — это не бесплатное увеличение всех работников. Перевод жителей в ремесленники должен создавать выбор: игрок может потерять рабочих для ферм и получить специалистов для фабрик.

### Критерий завершения

Игрок осознанно балансирует количество поселенцев и ремесленников.

---

## Этап 6. Полноценные производственные цепочки

**Приоритет: P0**

### Цель

Добавить взаимосвязанные цепочки с разной сложностью и назначением.

### Ветка 1. Картофель и чипсы

```text
Картофельная ферма
  → картофель
  → завод чипсов
  → чипсы
  → продажа / потребность / контракт
```

### Ветка 2. Пшеница и хлеб

```text
Пшеничное поле
  → пшеница
  → мельница
  → мука
  → пекарня
  → хлеб
  → потребление населением / продажа
```

### Ветка 3. Апельсины и сок

```text
Апельсиновая роща
  → апельсины
  → соковый завод
  → сок
  → цех розлива
  → бутилированный сок
  → продажа / контракт
```

### Ветка 4. Строительство

```text
Лесоруб → дерево → лесопилка → доски
Каменоломня → камень
Глиняный карьер → глина → кирпичный завод → кирпич
```

### Требования к конфигам

Каждый ресурс и здание должны описываться данными:

- id;
- категория;
- входы;
- выходы;
- длительность цикла;
- рабочая сила;
- стоимость строительства;
- условия открытия;
- базовая продажная стоимость;
- потребительская ценность;
- локализационные ключи по id.

### Балансовое правило

Чем длиннее цепочка, тем выше потенциальная прибыль, но тем больше:

- стоимость запуска;
- потребность в рабочих;
- складские риски;
- зависимость от промежуточных ресурсов;
- время окупаемости;
- требования населения.

### Критерий завершения

В игре есть минимум три осмысленные ветки, и каждая решает отдельную задачу.

## Этап 7. Управление производством

**Приоритет: P1, часть функций может попасть в P0**

### Задачи

- Добавить включение/выключение здания.
- Добавить приоритет распределения рабочей силы.
- Добавить приоритет потребления ресурсов.
- Добавить лимит автопродажи: продавать только излишки сверх заданного резерва.
- Добавить целевой запас ресурса.
- Добавить прогноз производства и потребления в минуту.
- Показать причину простоя здания.

### Состояния здания

Минимальный список:

```text
idle
running
waiting_for_inputs
waiting_for_workers
output_blocked
disabled
```

### Критерий завершения

Игрок понимает, почему производство остановилось, и может управлять дефицитами без микроменеджмента каждой секунды.

## Этап 8. Уровень города и открытия

**Приоритет: P0/P1**

### Цель

Не показывать весь контент сразу и создать понятные ближайшие цели.

### Условия уровня города

Уровень может зависеть от:

- общего населения;
- количества ремесленников;
- выполненных контрактов;
- построенных ключевых зданий;
- общего оборота;
- удовлетворённости населения.

### Принципы открытия

- Игрок всегда видит следующую цель.
- Новый уровень открывает 1–3 важных объекта, а не десять сразу.
- Заблокированный контент показывает условия открытия.
- Открытие сопровождается заметным, но дешёвым визуальным событием.

### Критерий завершения

Первая игровая сессия имеет последовательность целей, а не бесконечный пустой рост чисел.

---

## Этап 9. Контракты и задания

**Приоритет: P1**

### Типы контрактов

- поставить определённое количество товара;
- поддерживать производство в течение времени;
- построить заданное здание;
- достичь нужного населения;
- удовлетворить потребности класса;
- переработать, а не продать сырьё.

### Награды

- деньги;
- строительные ресурсы;
- временный буст;
- очки исследования;
- доступ к зданию;
- постоянный небольшой бонус.

### Правила

- Контракты не должны требовать просмотра рекламы.
- Rewarded-реклама может улучшить награду, но не заменять основной прогресс.
- Контракты должны направлять игрока в новые механики.

### Критерий завершения

У игрока всегда есть 1–3 конкретные цели на ближайшие 5–20 минут.

---

## Этап 10. Offline-прогресс

**Приоритет: P0**

### Цель

Возвращение в игру должно приносить ощутимую награду.

### Базовая модель

- Сохранять время последнего корректного сохранения.
- При загрузке рассчитывать прошедшее время.
- Ограничить базовый offline-прогресс, например двумя часами.
- Выполнить симуляцию безопасно, без миллионов последовательных tick.
- Учитывать входные ресурсы, склад, рабочую силу и потребление населения.
- Не создавать ресурсы из воздуха.
- Показывать отчёт: произведено, продано, потреблено, потеряно из-за простоя.

### Монетизация

Rewarded-реклама может:

- удвоить offline-доход;
- увеличить учитываемый период;
- временно расширить склад;
- компенсировать часть простоя.

### Безопасность

- Ограничить максимальный период.
- Проверять невалидные даты.
- Не доверять времени устройства без ограничений.
- Не допускать зависания UI при расчёте.

### Критерий завершения

Игрок получает честный и объяснимый результат отсутствия, а игра не зависает после долгого перерыва.

---

## Этап 11. Улучшения и исследования

**Приоритет: P1**

### Улучшения зданий

- скорость производства;
- объём за цикл;
- уменьшение потребности в рабочих;
- снижение расхода сырья;
- увеличение локального буфера;
- повышение качества/цены продукции.

### Исследования

Начать с небольшого дерева на 10–15 узлов.

Примеры направлений:

- сельское хозяйство;
- переработка;
- строительство;
- логистика;
- управление рабочей силой;
- торговля.

### Правила

- Улучшения должны иметь заметный эффект.
- Не добавлять десятки улучшений с бонусом 1%.
- Постоянные бонусы должны быть ограничены, чтобы не уничтожить баланс.

### Критерий завершения

Игрок может специализировать экономику и принимать долгосрочные решения.

---

## Этап 12. Удержание

**Приоритет: P1**

### Минимальный набор

- ежедневная награда;
- серия входов без жёсткого наказания за пропуск;
- достижения;
- недельные цели;
- уведомление о заполненном складе внутри игры;
- история последних важных событий;
- визуальные этапы развития города.

### Не использовать

- агрессивные таймеры страха упустить выгоду;
- блокирование базового прогресса рекламой;
- наказание за отсутствие;
- всплывающую рекламу каждые несколько минут;
- ложные скидки и тёмные паттерны.

### Критерий завершения

Игроку есть зачем вернуться завтра, даже если он не смотрит рекламу.

---

## Этап 13. Интеграция с Яндекс Играми

**Приоритет: P0**

### Архитектура

Создать узкий platform adapter. Локальная версия игры должна работать без iframe и SDK.

Пример интерфейса:

```ts
interface GamePlatform {
  init(): Promise<void>;
  ready(): void;
  getLanguage(): string | null;
  getPlayerData(): Promise<unknown | null>;
  setPlayerData(data: unknown): Promise<void>;
  showRewardedAd(): Promise<'rewarded' | 'closed' | 'error'>;
  showInterstitialAd(): Promise<'closed' | 'error'>;
  submitLeaderboardScore?(score: number): Promise<void>;
}
```

### Обязательные функции

- инициализация SDK;
- Loading API;
- язык платформы;
- пауза симуляции и звука во время рекламы;
- rewarded-реклама;
- interstitial только в естественных паузах;
- облачное сохранение с локальным fallback;
- корректная работа неавторизованного игрока;
- обработка ошибок SDK без падения игры.

### Правила рекламы

Rewarded-реклама только за добровольную награду:

- x2 offline-доход;
- временный буст производства;
- дополнительный контракт;
- ускорение строительства;
- бесплатный временный склад.

Interstitial-реклама:

- не во время активного клика;
- не в первые минуты игры;
- не чаще разумного лимита;
- только после завершённого действия или перехода этапа.

### Критерий завершения

Игра проходит локальный запуск и запуск внутри Яндекс Игр, а ошибка платформы не ломает основной игровой цикл.

---

## Этап 14. Аналитика и баланс

**Приоритет: P1**

### События

- начало игры;
- первая покупка;
- первая завершённая цепочка;
- первое улучшение населения;
- первый дефицит рабочих;
- первый возврат после offline;
- просмотр rewarded-рекламы;
- отказ от рекламы;
- достижение уровней города;
- длительность сессии;
- причина выхода, если её можно определить косвенно.

### Метрики

- completion первой цепочки;
- время до первой фабрики;
- время до первого ремесленника;
- D1/D3/D7 retention;
- средняя длина сессии;
- rewarded opt-in rate;
- доля игроков, застрявших из-за денег, ресурсов, рабочих или склада;
- доход на активного игрока;
- частота сброса прогресса.

### Правило

Не менять баланс только по ощущениям после релиза. Использовать данные и игровые симуляции.

---

## Этап 15. Prestige / новая эпоха

**Приоритет: P1 после стабильного основного цикла**

### Цель

Дать долгосрочную причину начать заново с постоянным преимуществом.

### Возможная модель

Игрок продаёт или преобразует первую компанию/город и получает очки влияния.

Очки влияния дают ограниченные постоянные бонусы:

- небольшой рост скорости;
- стартовые строительные ресурсы;
- дополнительный лимит offline;
- новые специализации;
- косметические варианты.

### Правила

- Prestige не должен появляться слишком рано.
- До первого prestige игрок должен увидеть все основные механики.
- Нельзя полностью обесценивать ранние цепочки.
- Бонусы должны иметь diminishing returns.

Механика считается завершённой только если:

- существует чистая доменная модель;
- UI не дублирует формулы;
- значения задаются конфигами;
- обработаны ошибки и блокирующие состояния;
- добавлены переводы;
- сохранения совместимы;
- добавлены тесты;
- массовые значения не вызывают зависаний;
- механизм работает на desktop и mobile;
- пользователь понимает состояние без чтения кода;
- проверки и production build проходят.

# Итоговое направление

Business Universe должна стать не простым кликером, где игрок бесконечно покупает одинаковые фермы, а компактной экономической стратегией:

```text
строительные ресурсы
  → жильё
  → население
  → рабочая сила
  → производство
  → удовлетворение потребностей
  → развитие населения
  → новые профессии и фабрики
  → более сложные товары
  → контракты, исследования и новая эпоха
```

Главная ценность игры — связанные системы. Каждое новое здание должно либо обеспечивать строительство, население, производство, потребление, торговлю или долгосрочное развитие. Не добавлять сущности, которые отличаются только названием, иконкой и цифрами.
